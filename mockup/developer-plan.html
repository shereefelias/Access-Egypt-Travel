Gallery Section Not Rendering on Frontend
Critical

The "Access Egypt Travel Gallery" section on the homepage shows only the title to public visitors. The gallery images appear in Elementor's edit mode (admin view) but are invisible on the live site. This is a known Elementor issue with three possible causes.

Step 1 — Try the fast fix first (Regenerate CSS):

WordPress Admin → Elementor → Tools → Regenerate CSS & Data → click the button

After regenerating, visit the homepage in an incognito window and check if the gallery appears. If yes, done. If no, continue to Step 2.

Step 2 — Check the gallery widget source:

Homepage → Edit with Elementor → click the gallery widget → left panel → Content tab → check "Source" setting

If the source is set to Dynamic (pulling from a field, post type, or media category): switch it to Static and manually select the gallery images from the Media Library. Dynamic sources require admin authentication in some configurations and will appear blank to logged-out visitors.

Step 3 — Check for JavaScript conflicts:

Open the homepage in an incognito window → Right-click → Inspect → Console tab. Look for any red JavaScript errors. Common culprits:

// These plugins frequently conflict with Elementor gallery scripts:
- WooCommerce (jQuery version conflicts)
- BABE booking plugin scripts
- Any slider or lightbox plugin loaded separately

// Quick test: temporarily deactivate plugins one by one while checking
// the frontend in incognito. The gallery appearing = that plugin is the conflict.

Step 4 — If conflict found, enqueue fix:

Add the following to the active child theme's functions.php or via Elementor → Custom Code:

// Defer conflicting scripts — adjust handle name to match actual conflict
add_filter( 'script_loader_tag', function( $tag, $handle ) {
    $defer = [ 'woocommerce', 'babe-front' ]; // add conflicting handles
    if ( in_array( $handle, $defer ) ) {
        return str_replace( ' src', ' defer src', $tag );
    }
    return $tag;
}, 10, 2 );

Step 1 — Load Leaflet into WordPress (functions.php)
Do this once — affects all pages

You need to tell WordPress to load the Leaflet CSS and JS files on every page. Open your theme's functions.php file.

Use a child theme. If you edit the parent theme's functions.php directly, your changes get wiped when the theme updates. Go to Appearance → Themes and check if a child theme is active. If not, create one first (or use a plugin like "Child Theme Configurator").
Appearance → Theme File Editor → select functions.php from the right panel

Scroll to the very bottom of the file and paste this block — do not delete anything that's already there:

// Load Leaflet map library on all pages
function aet_load_leaflet() {
    wp_enqueue_style(
        'leaflet-css',
        'https://unpkg.com/leaflet@1.9.4/dist/leaflet.css',
        [],
        '1.9.4'
    );
    wp_enqueue_script(
        'leaflet-js',
        'https://unpkg.com/leaflet@1.9.4/dist/leaflet.js',
        [],
        '1.9.4',
        true  // true = load in footer, not head
    );
}
add_action( 'wp_enqueue_scripts', 'aet_load_leaflet' );

Click Update File. Check the live site loads without errors (press F12 → Console tab — should be no red errors).

A breadcrumb is the small trail above the page title — Home › Tours › 6 Days Cairo & Luxor — that tells a visitor where they are and gives them a one-click way back up the tree. The mockups already show this on tour pages (.tour-crumb) and destination pages (.dest-crumb). This section is the part the plan was missing: how to actually build the visible trail in WordPress. (Elementor Pro has a Breadcrumbs widget, but the Rank Math method below is the recommended one — it keeps the visible trail and the Google schema in sync from a single source and won't be affected by the Pro/core version gap.)

Two jobs, one component. Breadcrumbs do two things at once: (1) on-page navigation — the visible trail; and (2) SEO — Google shows the trail in the result's URL line via BreadcrumbList structured data. The SEO/schema side is already covered in seo-plan-phase3.html → step 1E. Do not build the schema twice — if you use the Rank Math method below, the schema is generated automatically from the same trail.

Why Pro Is Stuck at 3.7.7

Elementor ships as two plugins: Elementor (free core) and Elementor Pro (the paid add-on that sits on top of core). The free core had auto-updates on and climbed to 3.21.1. Pro had auto-updates off and nobody ran its update manually, so it stayed at 3.7.7. The license never lapsed — the update was just never clicked.

The gap matters because Pro's widgets and Theme Builder call into the free core. As the versions diverge, widget rendering, CSS output, and performance features mismatch — which is the likely cause of the broken Contact hero and other glitches in the Live Audit.
