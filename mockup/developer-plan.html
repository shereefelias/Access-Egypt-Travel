Gallery Section Not Rendering on Frontend
Critical

The "Access Egypt Travel Gallery" section on the homepage shows only the title to public visitors. The gallery images appear in Elementor's edit mode (admin view) but are invisible on the live site. This is a known Elementor issue with three possible causes.

Step 1 — Try the fast fix first (Regenerate CSS):

WordPress Admin → Elementor → Tools → Regenerate CSS & Data → click the button

After regenerating, visit the homepage in an incognito window and check if the gallery appears. If yes, done. If no, continue to Step 2.

Step 2 — Check the gallery widget source:

Homepage → Edit with Elementor → click the gallery widget → left panel → Content tab → check "Source" setting

If the source is set to Dynamic (pulling from a field, post type, or media category): switch it to Static and manually select the gallery images from the Media Library. Dynamic sources require admin authentication in some configurations and will appear blank to logged-out visitors.

Step 3 — Check for JavaScript conflicts:

Open the homepage in an incognito window → Right-click → Inspect → Console tab. Look for any red JavaScript errors. Common culprits:

// These plugins frequently conflict with Elementor gallery scripts:
- WooCommerce (jQuery version conflicts)
- BABE booking plugin scripts
- Any slider or lightbox plugin loaded separately

// Quick test: temporarily deactivate plugins one by one while checking
// the frontend in incognito. The gallery appearing = that plugin is the conflict.

Step 4 — If conflict found, enqueue fix:

Add the following to the active child theme's functions.php or via Elementor → Custom Code:

// Defer conflicting scripts — adjust handle name to match actual conflict
add_filter( 'script_loader_tag', function( $tag, $handle ) {
    $defer = [ 'woocommerce', 'babe-front' ]; // add conflicting handles
    if ( in_array( $handle, $defer ) ) {
        return str_replace( ' src', ' defer src', $tag );
    }
    return $tag;
}, 10, 2 );

19. Elementor Pro Upgrade — Version Mismatch Fix

Current state is unstable. Free Elementor is at 3.21.1 and Elementor Pro is frozen at 3.7.7 — 14 versions behind. Pro stopped receiving updates because the license expired. The site is working by luck. Each new WordPress or plugin update widens the gap and increases the risk of a sudden break.
PluginCurrent VersionLatest AvailableStatus
Elementor (free) 3.21.1 4.1.3 Update available — do NOT update yet
Elementor Pro 3.7.7 Requires active license to see License expired — no updates received
Do NOT click "Update Now" on free Elementor until Pro is updated first. Jumping free to 4.1.3 while Pro stays at 3.7.7 is what actually breaks the site — they must be updated together in the correct order.

Why Pro Is Stuck at 3.7.7

Elementor Pro licenses are annual subscriptions. When the license expires, the plugin stays installed and working, but no further updates are delivered. The free Elementor plugin is separate and kept updating automatically — which is why there is now a 14-version gap between the two.

The gap matters because Elementor Pro's widgets and Theme Builder rely on the free plugin's core. As the versions diverge, widget rendering, CSS output, and performance features become increasingly mismatched.

What You Need to Do

1
Renew the Elementor Pro license
First

Go to my.elementor.com and log in with the account that originally purchased Pro. Renew the license (currently ~$59–99/year for a single site). Once renewed, WordPress will show an available update for Elementor Pro.

my.elementor.com → Subscriptions → Renew
If you don't have the original account credentials, check the original purchase email. The license is tied to an email address, not the WordPress site.
2
Take a backup before touching anything
Before updating

Before running any updates, take a full backup using one of these methods:

  • Elementor Export Kit (fastest — saves all Elementor data): Elementor → Tools → Export Kit → Export
  • Host-level backup: Most managed WP hosts (SiteGround, Kinsta, WP Engine) have one-click backups in their dashboard
  • UpdraftPlus plugin: Free, backs up files + database to Google Drive
A backup is not optional. Proceed to step 3 only after you have a confirmed backup you can restore from.
3
Update Elementor Pro first, then free Elementor
Order matters
Plugins → Installed Plugins → find Elementor Pro → click Update Now

Wait for Pro to finish updating completely. Then:

Plugins → Installed Plugins → find Elementor (free) → click Update Now
After both updates, go to Elementor → Tools → Regenerate CSS & Data immediately. This rebuilds all the CSS files with the new version and fixes most visual glitches.
4
Check these pages immediately after updating
Visual QA

Open each of these pages in a private/incognito browser window and check for layout issues:

5
Optional but recommended: Use a staging environment
Reduces risk to zero

If your host supports staging (SiteGround, Kinsta, WP Engine all do), run the update on a staging clone of the live site first. Fix anything that breaks there, then push to live. This is the zero-risk approach.

Host dashboard → Staging → Create Staging Site → clone live → run updates on staging
Staging sites are temporary — they're just a copy. Nothing you do there affects the live site.

Why This Is Worth Doing Before Phase 2 SEO Work

Elementor 3.10+ shipped "Improved Asset Loading" — a feature that stops loading Elementor's full CSS/JS bundle on pages that don't need it. On 3.7.7, every page loads the full bundle regardless. This is a direct, measurable hit to your Core Web Vitals (LCP and INP scores), which Google uses as a ranking signal. Completing the SEO plan (Phase 2) on top of slow Core Web Vitals is building on a weak foundation. Upgrading Elementor Pro is the single change with the highest SEO ROI.
